Overview

BBOS (Barakah Business Operating System) is a modular AI operating system built on Claude. It guides operators through a structured 9-stage pipeline — from initial intake to optimisation — using a governor-driven architecture that enforces stage isolation, truth-safe language, and integrity validation at every step.

The operating hierarchy is: Governor → Active Stage Skill → Specialist Skills → Validators → State Update. No stage work happens without the governor first confirming routing.

The dashboard visualises the live state of each project — its active stage, routing decision, integrity status, required inputs, and session history — drawing data from a paired Google Sheet updated after each session.

Routing status

After every session the governor issues one of four routing decisions. This should be recorded in the meta sheet's routing_status key.

advance
All stage criteria met, all validators passed, no blocking issues. The project may move to the next stage.
remain
Stage work is in progress. Continue working within the current stage on the next session.
patch
A specific named output needs revision. Use the PATCH command on the next session targeting only that section.
no_ship
A hard stop. Required inputs are missing, a blocking issue is unresolved, or a request would require guessing. Nothing ships until the condition is resolved. Record the reason in noship_1.
Glossary
Governor
The runtime controller that resolves the active command and stage, enforces stage isolation, loads permitted specialists and validators, and issues the routing recommendation. Nothing in BBOS runs without the governor first approving it.
Stage isolation
The hard rule that work in one stage cannot pull canon, outputs, or methods from another stage. The governor enforces this at every session. Violating stage isolation produces unreliable, cross-contaminated outputs.
NO_SHIP
A governor decision that halts all output. Triggered when required inputs are missing, a blocking issue is unresolved, stage isolation would be violated, or producing output would require guessing.
Example: "NO_SHIP — no approved OFR Asset Pack exists. Cannot begin Outreach stage."
Anchor discipline
Each stage has a set of approved anchors (e.g. @@BBOS_STAGE:OFR@@). The governor only allows the active stage's anchors to be used. Invoking an anchor from another stage is a violation.
G-label
A classification applied to result-implying language in BBOS copy. G1 = grounded certainty (proven, documented). G2 = conditional claim (likely, typical). G3 = aspirational/directional. G4 = visionary/hypothetical. Copy that uses G4 language as if it were G1 is called certainty theater and will fail the Truth Gate validator.
G1: "Clients in this niche typically see X — documented across 12 case studies." G4 (invalid as a promise): "You will achieve X."
Certainty theater
Language that presents aspirational or conditional outcomes as guaranteed facts. A Truth Gate validator failure. Must be patched before stage can advance.
Truth Gate Validator
A validator that checks every meaningful claim is G-labeled, every G1/G2 claim has a proof anchor, no certainty theater exists, scarcity is documented or absent, and no claim exceeds scope control.
Amanah Proof Audit
A QAL-stage check of the operator's claimed credentials, competence, and results against documented evidence. Items that cannot be proven are flagged as proof gaps and must be resolved before the offer can make those claims.
Barakah Health Index (BHI)
A 5-dimension assessment used in Stage 09_OPT measuring the alignment of business operations with principles of integrity, sustainability, and blessing (barakah). BHI-1 through BHI-5 cover different operational domains.
Canonical Metrics (CM-1–4)
The four core performance metrics tracked in Stage 09_OPT: CM-1 Qualified Outreach Conversion Rate, CM-2 Fit-to-Close Rate, CM-3 Client Milestone Completion Rate, CM-4 Unprompted Referral Rate.
VoC (Voice of Customer)
Verbatim language collected from real prospects or clients in Stage 03_STR. Used to seed positioning, hooks, and objection responses. BBOS requires a minimum of 15 verbatim phrases in the VoC Buying Language Bank before advancing STR.
Scope Map
OFR Asset 4. Documents exactly what is included and excluded in the offer. Scope exclusions must be explicit — not implied. Prevents scope creep, misaligned expectations, and guarantee disputes.
Asset Pack
The complete set of deliverables produced by a stage. Each stage has a defined asset pack with numbered items that must all be complete before the stage can advance. Partial packs do not qualify.
Spiritual annotation
An operator-facing spiritual framing of a stage asset, invoked only by the explicit ANNOTATE command. Never auto-generated. Maximum 2 sentences per asset, 1 operational implication. Cannot be used to reinforce urgency, scarcity, or closing pressure.
No-fit / DQ language
Scripted language for declining or disqualifying prospects who are not a good match. BBOS requires this language to be clear, respectful, and final — not ambiguous or pressure-inviting. Dignity must be preserved.
G-label system

Every result-implying claim in BBOS copy must carry a G-label. The label is recorded internally — it does not appear in client-facing copy. Its purpose is to ensure the operator and validator can audit whether each claim is supported by evidence.

LabelMeaningProof requirement
G1Grounded certainty — documented, proven, repeatableMust have a named proof anchor or case study
G2Conditional claim — likely, typical, common outcomeMust reference pattern evidence or documented probability
G3Aspirational/directional — intended, designed forMust be framed as aspiration, not commitment
G4Visionary — hypothetical, potential, possibleMust be clearly framed as possibility only

G-flags in the dashboard indicate claims that are missing a label, using the wrong label for their proof level, or presenting G3/G4 language as if it were G1/G2 (certainty theater). These must be resolved before stage advance.

Validators

Validators are independent of generators. A stage cannot advance unless all required validators pass. Validator results should be recorded in the meta sheet's validator_N keys using the format validator-name::pass or validator-name::fail.

Spiritual layer

The spiritual layer in BBOS is operator-facing only and explicitly invocation-only. It is never generated automatically. Rules:

  • Only invoke with the ANNOTATE command — never without it
  • One annotation per major asset, maximum 2 sentences
  • Must include one specific operational implication
  • Never use spiritual language to reinforce urgency, scarcity, or closing pressure
  • If the active stage lacks a spiritual anchor, the result is NO_SHIP — not an improvised annotation
Sheet setup

Each project requires a Google Sheet with two tabs: meta and checklist. Use the BBOS Sheet Creator Apps Script to generate this automatically.

To publish the sheet for dashboard access: File → Share → Publish to web. Publish each tab individually as CSV. The dashboard uses the gviz/tq endpoint — this requires CSV publication, not "Web page" format.

Common error: Using the long /d/e/2PACX-… URL from the publish dialog instead of the Sheet ID from the edit URL. The dashboard needs the Sheet ID, not the published page URL.

FAQ
Why does the dashboard show "Failed to fetch"? ▾
The most common cause is publishing the sheet as "Web page" instead of "CSV". Go to File → Share → Publish to web → select your sheet tab from the first dropdown → select CSV from the second dropdown → Publish. Repeat for both the checklist and meta tabs. Also check that the Sheet ID in CONFIG is the short alphanumeric ID from the edit URL, not the long /d/e/ published URL.
Can a project skip a stage? ▾
No. Stage isolation is a hard rule. Each stage has defined prerequisite inputs from the stage before it. Skipping a stage means those inputs don't exist, which will trigger NO_SHIP when the skipped stage's downstream work is attempted. The only exception is that stages 07_FUL, 08_RET, and 09_OPT can be worked concurrently with the sales pipeline under specific conditions defined in the governor rules.
What does it mean when last_validated is behind last_session? ▾
It means work has been done in sessions after the last validator run. This is a yellow flag — not a block — but it means the integrity status shown may be based on older work. Run validators again before treating the current state as clean.
When should I use PATCH vs starting a new ASSEMBLE session? ▾
Use PATCH when the routing decision is specifically "patch" and a named section has been identified for revision. PATCH touches only that section — everything else is preserved. Use ASSEMBLE when building the full asset pack from scratch or doing a substantive rebuild. Never use ASSEMBLE to "touch up" a single item — that is PATCH work and creates unnecessary regeneration risk.
Can ANNOTATE be invoked for any stage? ▾
No. Only stages that have a defined spiritual anchor in their skill file support ANNOTATE. If you invoke ANNOTATE on a stage without an anchor, the result is NO_SHIP — not an improvised annotation. Check the stage skill file for the presence of @@SECTION:SPIRITUAL_ANCHOR@@ before invoking.
How do I clear a G-flag from the dashboard? ▾
Resolve the issue described in the flag (fix the claim, add the proof anchor, remove the certainty theater), then delete or blank out the corresponding gflag_N row in the meta sheet. Do not just overwrite it with a note — remove it entirely so the dashboard no longer counts it as an active flag.
What is the difference between a blocking issue and a NO_SHIP? ▾
A blocking issue is an operator-recorded condition that prevents progress — it lives in the meta sheet and is visible on the dashboard. A NO_SHIP is a governor decision issued during a Claude session when a hard stop condition is met. A blocking issue often causes the next session to return NO_SHIP, but they are recorded separately. Always log both: the blocking issue in blocking_issue_N and the NO_SHIP event in noship_N with a reason and date.
